Tarjuman — Speech-to-Text & Text-to-Speech Provider Analysis

Deepgram · OpenAI · Google Gemini  |  Recommendation for a real-time Arabic→English khutbah transcription app  |  Prepared June 14, 2026
Bottom line. Your current stack is already the right one — keep it. The best realtime Arabic STT is Deepgram Nova-3, and you correctly run a second engine (OpenAI Whisper) alongside it for accuracy. The best TTS for your 30+ target languages is OpenAI, with the browser Web Speech API as the free fallback. Don't add Gemini to the realtime path (it's an LLM doing audio, not a low-latency STT engine). The only changes worth making now are two drop-in model upgrades and a translation retry — and even those come after you've field-tested Deepgram in a real masjid. Redundancy is insurance; don't pay the premium before you have users to protect.

More than one provider — concurrently?
STT: yes, and you already do it right. Deepgram realtime + Whisper second-opinion is a deliberate dual-engine design for accuracy. Keep two — don't add a third concurrent engine.
TTS: no. One cloud primary + Web Speech fallback. Running multiple cloud TTS at once just burns money.
More than one as fallback?
STT: one realtime failover (OpenAI) is enough. TTS: the cloud→Web-Speech two-tier is enough.
The gap nobody asked about: translation has no fallback — and translation is the product. A transient Anthropic blip currently errors the segment. A simple retry / secondary is the cheapest, highest-value resilience win you can make.
What most people miss. Redundancy is insurance, and insurance has a premium — complexity, cost, and forever-maintenance. With zero validated users and a core that hasn't been field-tested in a real masjid, building elaborate multi-provider failover is premature. Validate Deepgram in the real environment first. Don't build a Deepgram failover before you've proven Deepgram itself works. An outage today hurts ~0 users; resilience engineering earns its keep once it protects someone.

What I'd actually do — in priority order

  1. nowField-test Deepgram first. Everything above assumes the primary is good enough in a real room. Prove that before changing anything (see docs/field-test-protocol.md).
  2. cheap winTwo drop-in model upgrades: Whisper → gpt-4o-transcribe (STT 2nd opinion) and tts-1-hd → gpt-4o-mini-tts (TTS). Better quality, lower cost, ~one line each. Do these after the field test so you don't change two variables before measuring the first.
  3. cheap winAdd a translation retry/secondary. Translation is the product and has no fallback today. One retry (or a cached/secondary model) is high-value insurance.
  4. when you have usersRealtime STT outage failover to OpenAI gpt-realtime-whisper, triggered only when Deepgram can't reconnect.
  5. skip for nowGemini on the realtime path; Deepgram Aura-2 TTS; any concurrent third engine.
